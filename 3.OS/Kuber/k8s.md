# Что происходит когда вводишь команду k8s
---
```bash
kubectl create deployment nginx --image=nginx --replicas=3
```
Появляется 3 nginx пода, распределенных по всем нодам
Обрабатывает развертывание рабочих нагрузок в инфраструктуре при помощи API

## Kubectl
---
- Валидация и генерация

Валидация на стороне клиента (Запросы с ошибками: неправильные ресурсы, неверные имена образов, ...) Такие запросы выведут ошибку и не отправят запрос на kubeAPI.
Это улучшает взаимодействия с системой за счет снижения нагрузки.

После варидации просиходит сборка HTTP запроса для отправки на Kube-apiserver
Все попытки доступа или изменения состаяния происходят через kube-apiserver, который взаимодействует

Kubectl использует генераторы для сериализации.

В команде `kubectl run` можно обращаться к нескольким ресурсам.
`kubectl run` определяет тип ресурса - автоматически, если тип ресурса не указан явно через `--generator`

Примеры:
`--restart-policy=Always` - Deployments
`--restart-policy=Never` - Pods


Kubectl также выяснит, нужно ли запускать другие действия.
Kubectl не просто слепо отправляет запрос в кластер. Он выполняет дополнительную логику на клиенте.

А) Запись команд (для откатов или аудита) — --record (устарело)
Раньше был флаг --record. Если ты его указывал:

```bash
kubectl run my-nginx --image=nginx --record
```
kubectl добавлял в аннотации создаваемого ресурса команду, которой он был создан.
Это было полезно, чтобы посмотреть, кто и что запускал, а также при выполнении kubectl rollout history можно было увидеть, какая команда привела к этому изменению.
(Сейчас вместо --record рекомендуется использовать декларативное управление файлами и системами CI/CD).

Б) Пробный прогон (Dry Run) — --dry-run
Это очень полезная штука, которую используют постоянно.

Флаг --dry-run говорит: «Не отправляй это по-настоящему в кластер, просто покажи, что бы произошло».

--dry-run=client:
Проверка клиенте.
Kubectl сформирует запрос, но не пошлет его на сервер.
Самый безопасный способ проверить синтаксис.

--dry-run=server:
Запрос уходит на сервер, сервер проверяет права доступа и валидирует схему, но не сохраняет объект (не создает его по-настоящему).


Пример использования:
Хочешь создать YAML-файл для пода, но не создавать его в кластере.

```bash
kubectl run my-pod --image=nginx --restart-policy=Never --dry-run=client -o yaml > my-pod.yaml
```
Что происходит:

kubectl определяет по --restart-policy=Never, что это Pod.

Видит флаг --dry-run=client.

Не обращается к кластеру.

Формирует YAML-представление этого пода в памяти.

Выводит его на экран (или в файл через >).
