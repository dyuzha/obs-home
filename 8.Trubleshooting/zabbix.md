# Zabbix
---
Система мониторига инфраструктуры.

1. Сбор данных
2. Орбработка и хранение
3. Анализ и оповещение

## Основные сущности
---
**Host**  (любой объект который мониторится)
Имеет:
- интерфейсы (IP/DNS)
- элементы мониторинга
- триггеры
- шаблоны


**Item**  (элемент данных) - конкретная метрика сбора Zabbix
Например:
- CPU load
- Свободная память
- ping
- HTTP response time
- размер файла
- статус сервиса

**Trigger** - логическое условие, определяющие проблему
Например:
- CPU load > 5
- free disk < 10%
Если условие выполняется -> Создается Event, в интерфейсе появляется Problem

**Event** / **Problem**
Например:
Hight CPU laoad on server1

**Template** - Набор готовых настроек мониторинга
 Включает:
 - items
 - triggers
 - graphs
 - discovery rules

**Item tags** раньше - *Application* (depricated)
 Раньше items можно было группировать в Applications:
 - CPU
 - Memory
 - Network

**Graph** - визуализация данных
Графики строятся из **items**


**Dashboard** - экран наблюдения
Можно разместить:
- графики
- проблемы
- статус хостов
- SLA
- карты


## Методы сбора данных
---

**Zabbix Agent**
Работает в 2-ух режимах:
- Passive checks (Zabbix сам опрашивает агент)
```bash
Server -> Agent
```

- Active checks (Agent отправляет данные)
```bash
# Удобно через Nat
Server <- Agent
```

**Agentless мониторинг**
Мониторинг без агентов
- SNMP - для сетевого оборудования
- ICMP
- HTTP checks (проверка веб-сайтов)
- SSH / Telnet (выполнение команд)
- External scripts (кастомные скрипты)


## Discovery
---
Автообноружение - **Zabbix** умеет сам находить объекты
- Netwokr discovery -  Сканирует сеть
- Low Level Discovery (LLD) - Автоматическое создание items


## Proxy
---
Нужен для распределенного мониторинга

- Мониторинг удаленных серверов
- Снижение нагрузки
- Буферизация данных

Proxy - временно хранит данные, если сервер недоступен


## Allerting (уведомления)
---
Срабатывает trigger -> Запускается Action

Action может:
- отправить email
- TG сообщение
- Slack
- webhook
- выполнить скрипт
